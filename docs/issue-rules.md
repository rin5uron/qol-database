# issue 作成ルール

## 作成方法

- issue は **CLI 経由**で作成する。`gh issue create --title "..." --body "..."` を使う。
- GitHub の Web から新規作成するのではなく、リポジトリ内で `gh` コマンドで作成する。

---

## タイトル

- **先頭に【】で種別を入れる。** 例：【課題】【エラー】【学び】【ログ】【方針】。
- **日付はタイトルに入れない。** 日付は **本文の最初** に1行で書く。**右寄せ**で `2026-02-18` の表記だけ（「日付:」などは付けない）。右寄せは `<p align="right">2026-02-18</p>` のように書く。
- **長さの目安:** できるだけ短く。議事録は「[議事録] テーマ（キーワードのみ）」にし、詳細は本文に。目安 50 字以内。
- **【学び】で一緒に学んだこと:** タイトルには学んだテーマのワードを並べておく。例：【学び】Cloudflare・被リンク・API・タイムスリープ
- **【エラー】の書き方:** エラー名やキーワードを端的に。例：【エラー】PyVirtualDisplay use_2to3・A8 TimeoutException

---

## 種別とラベル（タグ）

- **ラベルは英字で1つ付ける。** 種別に応じて下表のラベルを使う。カタカナ・日本語のラベルは使わない（例: roadmap のみで「ロードマップ」は作らない）。
- **既存 issue に旧ラベルが付いている場合は、編集時に英字ラベルへ寄せる。** 例: `アイデア` → `policy` または `roadmap`、`学び` → `study`
- **ラベルは原則1つに絞る。** 迷ったら「その issue を将来どう使うか」で決める。

| 種別 | ラベル | 意味 |
|------|--------|------|
| 【課題】 | task | やること・ToDo。実施記録はその issue の本文やコメントに書く。 |
| 【エラー】 | bug | エラー・バグの記録と対応。発生したエラーとその解決方法を記録する。 |
| 【方針】 | policy | ルール・決めごと（「こうする」と決めたこと）。 |
| 【ログ】 | log | 記録用の issue。やったこと・起きたことを残すだけ。やるタスクはない。 |
| 【学び】 | study | 学んだことのまとめ。 |
| [ロードマップ] | roadmap | フェーズやマイルストーンの計画。 |
| [議事録] | meeting | 打ち合わせの記録。 |

- **作業した人・担当者は assignees（担当者）で表す。** ラベルではなく GitHub の Assignees に自分や一緒にやった人を追加する。例: `gh issue edit <番号> --add-assignee @rin5uron`
- **お互いの名前はアカウント名で呼ぶ。** issue・コメント・ドキュメント内では @rin5uron、@dataanalytics2020 のように GitHub アカウント名を使う。

**作成例（ラベル・assignee 付き）**

```bash
gh issue create --title "【学び】〇〇について" --body "<p align=\"right\">2026-02-18</p>\n\n（本文）" --label study --assignee @rin5uron
```

---

## 「上書きして」のとき

- **「上書きして」と言われたときは issue の本文（body）を編集する。** コメント欄には書かない。

---

## close / 継続の判断

- **実施完了を報告するだけの issue は閉じる。** 例: 「作成完了」「反映完了」「記録済み」。
- **参考URLやメモだけの issue は、本文を必要な場所へ転記できたら閉じる。**
- **まだ意思決定や実作業が残る issue は閉じない。** 完了条件が残っているなら open のままにする。
- **古い issue を閉じるときは、必要なら最後に1コメントで理由を書く。** 例: 「内容を docs に転記したため close」。
- **旧ルールで作られた issue でも、再利用するなら本文・タイトル・ラベルを整えて使ってよい。**
